T2

Advance rule — "best of Nth place" Effort ~6–10h Risk: กลาง–สูง schema + algo + UI

ปัญหาปัจจุบัน

ตอนสร้าง knockout ทุกกลุ่มส่งตัวเข้ารอบ เท่ากันหมด = top-advance_count ต่อกลุ่ม (seedsFromStandings(standings, advanceCount) ที่ matches.ts:659 / classes.ts:842 แค่ .slice(0, count) ต่อกลุ่ม ไม่มีการเทียบข้ามกลุ่ม).

เลยทำ bracket ที่ผู้เข้ารอบ ≠ ทุกกลุ่มเท่ากันไม่ได้ — เช่นอยากได้ 16 ทีมเข้ารอบ จาก 6 กลุ่ม (กลุ่มละ 4 คู่). Top 2 ทุกกลุ่ม = 12 ขาดอีก 4. รูปแบบมาตรฐาน (เหมือน World Cup / รายการแบดหลายรายการ) คือเอา "อันดับ 3 ที่ดีที่สุด 4 กลุ่ม" มาเติม — ตอนนี้ทำไม่ได้.

สิ่งที่ต้องทำ

  • เพิ่มคอลัมน์ tournaments.advance_rule jsonb (และ tournament_classes.advance_rule สำหรับ competition)
  • แก้ seeding ให้รวม 2 ชั้น: (1) auto-per-group top-N เหมือนเดิม + (2) "best rest" — เก็บแถวอันดับเดียวกันจากทุกกลุ่ม มาเรียง tiebreak แล้วตัด M ตัวบน
  • วาง seed ข้ามกลุ่มไม่ให้คู่กลุ่มเดียวกันเจอกันรอบแรก
  • vitest: aggregate ranking + tiebreak + placement
  • UI: ช่องตั้ง rule ใน Settings / สร้าง bracket
// รูปแบบ advance_rule jsonb ที่เสนอ
{
  "auto_per_group": 2,   // ทุกกลุ่มส่ง top 2
  "best_rest": { "rank": 3, "take": 4 }  // + อันดับ 3 ที่ดีสุด 4 กลุ่ม
}
// → target bracket = 6×2 + 4 = 16 (KO16)
ตัวอย่าง: 6 กลุ่ม กลุ่มละ 4 คู่. Top 2 ทุกกลุ่ม = 12 ผ่านอัตโนมัติ. เหลือสล็อต 4 → เทียบ คู่อันดับ 3 ของทั้ง 6 กลุ่ม ด้วย tiebreak เดิม (leaguePoints → pointDiff → pointsFor):
คู่ (อันดับ 3 ของกลุ่ม)Pts+/−PFผล
3rd กลุ่ม A6+852ผ่าน #1
3rd กลุ่ม D6+548ผ่าน #2
3rd กลุ่ม B6+544ผ่าน #3
3rd กลุ่ม F4+240ผ่าน #4
3rd กลุ่ม C4−138ตกรอบ

ได้ครบ 16 → สร้าง KO16. (ปัจจุบันได้แค่ 12 หรือต้องตั้ง advance_count=3 ซึ่งจะกลายเป็น 18 ไม่ลงตัว power-of-2)

T3

Tournament level → levels table Effort ~6–8h Risk: กลาง (มี backfill) migration + UI

ปัญหาปัจจุบัน

ฝั่ง tournament เก็บ level เป็น free-text: team_players.level (text), pairs.pair_level (text = ผลรวม). Division แบ่งโดย parseFloat(pair_level) เทียบ pair_division_thresholds[]. Free-text เลย drift ได้ง่าย — พิมพ์ "3,5" / "N" / "3.5 " (เว้นวรรค) / "มือ N" ทำให้ parseFloat ได้ NaN หรือเลขผิด → คู่ตกไป Division ผิด หรือ null.

ฝั่ง club ทำเสร็จแล้ว — club_players.level_id FK → levels (real numeric สำหรับคำนวณ + label สำหรับแสดง). T3 = ยกแบบเดียวกันมาฝั่ง tournament.

สิ่งที่ต้องทำ

  • เพิ่ม team_players.level_id FK → levels (nullable, ระหว่าง migrate)
  • backfill: map text เดิม → level_id ที่ตรง (real = parseFloat); ค่าที่ map ไม่ได้ flag ไว้แก้มือ
  • pair_level คำนวณจาก real ของผู้เล่น 2 คน (numeric แท้ ไม่ใช่ string sum)
  • Division compare ใช้ real โดยตรง — เลิก parseFloat(text)
  • UI: เปลี่ยน level input เป็น Select จาก levels (เหมือน club)
ตัวอย่าง drift ที่หายไป: ผู้เล่น 3 คน "ระดับ N" แต่กรอกต่างกัน:
กรอก (ตอนนี้ free-text)parseFloat ได้หลัง T3 (level_id→real)
"N"NaNreal 3 · label "N"
"3.5"3.5real 3.5 · label "N+"
"3,5"3 (ผิด!)เลือกจาก list — ผิดไม่ได้
pair_level = real(A) + real(B) คำนวณตรง → threshold เทียบแม่น, ไม่มีคู่หล่นผิด Division.

ไฟล์กระทบ

T4

Collapse divisions — หน้าแข่ง Effort ~2–3h Risk: ต่ำ (UI ล้วน) frontend

ปัญหาปัจจุบัน

หน้าแข่ง (knockout-stage.tsx / pair-stage.tsx) เรนเดอร์ทุก Division กางหมดพร้อมกัน. รายการที่มี 6–8 Division → scroll ยาวมากบนมือถือ กว่าจะถึง Division ล่างสุด.

สิ่งที่ต้องทำ

  • ห่อแต่ละ Division ใน collapsible (accordion / <details>)
  • หัวข้อโชว์สรุปย่อ: ชื่อ Division + จำนวนแมตช์ + champion (ถ้ามี)
  • จำ state เปิด/ปิด (เปิด Division แรกค้างไว้ default)
  • ไม่แตะ logic — แค่ wrap การแสดงผล
ก่อน: 8 Division กางหมด = หน้ายาว ~8 จอ 
▼ Division 1 — น็อคเอ้า (8 matches, champion: คู่ A/B)
   [ ...bracket เต็ม... ]
▼ Division 2 — น็อคเอ้า (8 matches)
   [ ...bracket เต็ม... ]
   ... อีก 6 division กางหมด
หลัง T4: ยุบเหลือหัวข้อ คลิกกางทีละอัน

   [ ...bracket... ]
▶ Division 2 — 8 matches            (ยุบ — คลิกกาง)
▶ Division 3 — 6 matches
▶ Division 4 — 6 matches · 🏆 คู่ X/Y
   ... สั้นลง เห็นภาพรวมทันที

T5

Granular queue Realtime Effort ~8–12h Risk: สูง (race / state) realtime + optimistic UI

ปัญหาปัจจุบัน

tournament-live-wrapper.tsx subscribe postgres_changes แล้ว debounced router.refresh() (setTimeout(..., REFRESH_DEBOUNCE_MS) 400ms). ทุกครั้งที่มี match เปลี่ยน (บันทึกผล/เริ่ม/เปลี่ยนสนาม) = refetch RSC ทั้งหน้า — re-render ทุก tab/ทุกตาราง แม้เปลี่ยนแค่ row เดียว. หนัก + กระพริบ ตอนแข่งจริงหลายสนามพร้อมกัน.

สิ่งที่ต้องทำ

  • flag queue_payload_sync ใน settings (เปิด/ปิดได้, default off ตอนแรก)
  • ใช้ payload จาก postgres_changes โดยตรง (matches มี REPLICA IDENTITY FULL อยู่แล้ว → payload มีทุกคอลัมน์)
  • patch เฉพาะ match row ที่เปลี่ยนเข้า local state — ไม่ refetch ทั้งหน้า
  • optimistic UI: กดเริ่ม/จบ → อัปเดต UI ทันที แล้วค่อย reconcile กับ payload จริง
  • handle INSERT / UPDATE / DELETE + ลำดับ event + กัน stale (เทียบ updated_at/ค่าใหม่)
  • fallback กลับ router.refresh() เมื่อ flag off
ตัวอย่าง: 4 สนามแข่งพร้อมกัน, กรรมการสนาม 2 กดจบแมตช์ #7
ขั้นตอนนี้หลัง T5
triggerUPDATE match #7UPDATE match #7
ฝั่ง viewerrefetch ทั้งหน้า (ทุก match, ทุก tab)patch แค่ card #7
re-renderทั้ง tree + dnd-kit re-init1 row
อาการกระพริบทั้งจอ ~400ms debounce#7 เปลี่ยนสถานะนิ่ง ทันที
ผลรวม: ลด network + ไม่กระพริบ + queue ใช้ลื่นขึ้นตอน peak หลายสนาม.

หมายเหตุความเสี่ยง: optimistic UI + realtime patch เปิดช่อง race (event มาสลับลำดับ, optimistic vs payload ขัดกัน, dnd reorder ชน Realtime update). ต้องทดสอบ live หลายสนามจริง — static check จับไม่ได้.
