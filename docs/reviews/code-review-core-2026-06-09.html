P0🔐 securitysrc/lib/actions/matches.ts:1106

1. recordMatchScoreAction is a cross-tournament IDOR — match write not scoped to tournamentId

Failure scenario

The action authorizes via assertCanEdit(input.tournamentId, ...) but then fetches the match with `sb.from('matches').select('*').eq('id', input.matchId).single()` (line 1106) — NO tournament_id filter — and writes via RPC record_match_score, which (verified in live DB) takes only p_match_id and has no tournament_id parameter. input.tournamentId is used ONLY for the authz check, audit, revalidate, and settings; it never constrains the write target. Attack: any authenticated user creates their own tournament (they become owner, so assertCanEdit passes for THAT id), then calls recordMatchScoreAction({ matchId: <victim tournament's match UUID>, tournamentId: <attacker's own tournament>, games: [...] }). The score, winner_id, and (for knockout) bracket advancement are written to the victim's match. This is a global match-write / result-tampering bypass across every tournament in the system.

Suggested fix

After fetching the match, assert `match.tournament_id === input.tournamentId` and return an error if not (the select already returns all columns). Better: add a `p_tournament_id` parameter to the record_match_score RPC and scope the `select ... for update` and final UPDATE with `AND tournament_id = p_tournament_id`, returning match_not_found when it doesn't match.

P1🗄 data-integritysrc/lib/actions/matches.ts:1491

2. resetMatchScoreAction corrupts another tournament's group standings before the scoped RPC guard fires

Failure scenario

resetMatchScoreAction fetches the match by id only (line 1488, no tournament filter), then at lines 1491-1495 calls reverseGroupTeamStandings(match.group_id, match.team_a_id, match.team_b_id, ...) using the fetched match's OWN group/teams. Only AFTERWARD does it call the reset_match_score RPC, which (verified in live DB) is correctly scoped (WHERE id=p_match_id AND tournament_id=p_tournament_id). Attack/mistake: caller passes matchId = a completed team-mode GROUP match belonging to tournament Y, tournamentId = attacker's tournament X (assertCanEdit passes for X). reverseGroupTeamStandings runs first and decrements Y's group_teams wins/draws/losses/points_for/points_against; then the RPC matches 0 rows and raises match_not_found, so the action returns an error and the match itself is never reset. Net effect: repeatable corruption of another tournament's standings (clamped at 0 by Math.max), with no successful reset to show for it.

Suggested fix

Verify `match.tournament_id === tournamentId` immediately after the fetch on line 1488 (return error otherwise) so reverseGroupTeamStandings never runs on an out-of-tenant match. The RPC is already scoped; the pre-RPC standings reverse is the leak.

P1🗄 data-integritysrc/lib/actions/matches.ts:1160

3. recordMatchScoreAction has no completed-match guard — re-recording double-counts group standings

Failure scenario

recordMatchScoreAction never checks match.status. record_match_score RPC unconditionally sets status='completed' and overwrites games. For a team-mode group match, lines 1160-1162 then call updateGroupTeamStandings which ADDS wins/draws/losses/points_for/points_against onto the existing group_teams row WITHOUT first reversing the prior result. The UI (match-row.tsx) only shows a reset button on completed matches, so the normal path is gated — but the server action has no guard, so any direct/replayed invocation (or a double-submit before the row flips to completed) on an already-completed team-group match runs updateGroupTeamStandings a second time and double-counts that team's standings (e.g. a 2-0 win recorded twice gives the winner 2 wins and 4 points-for from one real match), corrupting standings and downstream knockout seeding. Pair matches are exempt (guarded by !match.pair_a_id on line 1160).

Suggested fix

Reject re-recording in the action: after fetching the match, if match.status === 'completed' return an error (require explicit reset first), OR make updateGroupTeamStandings a delta that first reverses the match's previously-stored games. Enforce idempotency in the RPC by only applying when status <> 'completed'.

P1🔐 securitysrc/lib/actions/tournaments.ts:352

4. addTeamPlayerAction never verifies teamId belongs to the asserted tournamentId (IDOR)

Failure scenario

Attacker is a co-admin (or owner) of tournament X (passes assertCanEdit(X)). They call addTeamPlayerAction(teamId=<a team that belongs to victim tournament Y>, input, tournamentId=X). assertCanEdit only checks rights on X; the insert writes team_id=Y-team with no check that Y-team.tournament_id === X. A player row is injected into another tournament's team. Same defect in removeTeamPlayerAction (line 387: deletes team_players by id with no tournament/team ownership check) and deleteTeamAction (line 330: deletes teams by id with only assertCanEdit(tournamentId)). Contrast toggleTeamPlayerCheckInAction (line 818-825) which DOES re-fetch and verify team.tournament_id === input.tournamentId, proving the guard is intentional elsewhere and merely missing here.

Suggested fix

Before the write, fetch the team (addTeamPlayer/deleteTeam) or the player's team (removeTeamPlayer) and assert its tournament_id === the asserted tournamentId, returning an error if not — mirror the pattern already used in toggleTeamPlayerCheckInAction.

P1🗄 data-integritysrc/lib/actions/tournaments.ts:568

5. importPairsCsvAction maps players by csv_id alone — same csv_id on two teams picks the wrong player

Failure scenario

csv_id has no unique constraint (verified: team_players has only a PK on id, no unique index on csv_id). The query at line 546-549 fetches ALL team_players with non-null csv_id (no tournament filter), and playerByCsvId.set(p.csv_id, ...) at line 571 keys solely on csv_id. If two players on different teams within the SAME tournament share a csv_id (e.g. both teams used '1a'), the later row overwrites the earlier in the map. A pair CSV row referencing id_player_1='1a' then resolves to whichever team's player was inserted last, building a pair with the wrong player (and p1.teamId/p2.teamId mismatch may also wrongly skip valid pairs at line 610).

Suggested fix

Key playerByCsvId by a composite that disambiguates within the tournament — but since csv_id is meant to be the cross-team CSV key, instead detect duplicate csv_id within the tournament's players and return an error (or require csv_id global-per-tournament uniqueness) rather than silently last-write-wins. At minimum add a partial unique index on (team_id, csv_id) and resolve players via team context from the pair row.

P1🔐 securitysrc/lib/auth/session.ts:24

6. Signed session token has no expiry/iat — captured cookie is valid forever, no revocation

Failure scenario

SessionPayload is {profileId, displayName, pictureUrl, isGuest} with no `exp`/`iat`. `encode()` (line 24) signs only that, and `decode()` (line 29) only checks the HMAC. The 30-day `MAX_AGE` (line 5) is solely a cookie attribute the browser enforces — the server accepts any validly-signed token indefinitely. If a `bc_session` value leaks (shared device, proxy log, XSS-adjacent disclosure, backup), an attacker replays it months/years later and is fully authenticated as that profile; because SESSION_SECRET is shared and static, there is no per-session revocation, no way to invalidate one stolen token, and `clearSession()` only deletes the victim's own browser cookie. Owners/co-admins have full mutate rights, so this is account takeover with an unbounded window.

Suggested fix

Embed `iat` (and optional `exp`) in the signed payload; in `decode()` reject when `now - iat > MAX_AGE`. For revocation, mix a per-profile token-version/secret-salt into the HMAC key so bumping it invalidates all outstanding tokens for that user. Re-issue on a sliding window.

P1🐛 correctnesssrc/lib/tournament/bracket-visual.ts:47

7. buildVisualBracket drops lower-bracket matches: single-elim halving geometry applied to double-elim lower section

Failure scenario

buildVisualBracket(matches, "lower") is called for every double-elimination tournament (admin /tournaments/[id]/bracket page line 59, public /t/[token]/bracket line 93). The lower bracket's round sizes are [n/4, n/4, n/8, n/8, ...] (see buildDoubleBracket lowerRoundSizes, bracket.ts:120-125) — consecutive rounds are EQUAL size, not halving. But line 47 computes slotCount = Math.round(firstRoundCount / 2^idx), assuming strict halving, and line 51 only places matches where i < slotCount (`rMatches.forEach((m,i)=>{ if (i<slotCount) slots[i]=m })`). For an 8-entrant double-elim: lower round idx=1 has 2 real matches but slotCount evaluates to round(2/2)=1, so exactly 1 real, playable lower-bracket match is silently sliced out of both the admin and public bracket pages and never rendered. The bug worsens for n=16 (more consolidation rounds collapse). Secondary effects from the same root cause: slotHeight doubling (line 48) and the single-elim roundLabel teamsInRound math (line 55, used at line 19) are both meaningless for the lower bracket.

Suggested fix

Do not derive slotCount from single-elim halving for the lower section. Drive the row count from the actual data — e.g. slotCount = rMatches.length (so no match is ever dropped), and compute slotHeight/labels per-section instead of assuming bracketSize = firstRoundCount*2. The strict-halving path is only valid for section==="upper". Lower-bracket vertical alignment is cosmetic and can be handled separately; never slice real matches off.
