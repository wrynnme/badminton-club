สรุปสั้นที่สุด

เดิมระบบตรวจสลิปจ่ายเงินใช้ “คีย์กลาง” อันเดียวทั้งเว็บ — รอบนี้เปลี่ยนเป็น ตั้งค่าได้รายก๊วน: ก๊วนไหนอยากให้ตรวจสลิปอัตโนมัติก็ไปสมัครบริการ (SlipOK/EasySlip) มาเอง แล้วใส่กุญแจของตัวเอง ก๊วนที่ไม่อยากยุ่งก็ใช้แบบ “เจ้าของกดยืนยันเอง” (ค่าเริ่มต้น) งานนี้ผ่านการตรวจคุณภาพครบทุกด่าน ทดสอบบนเบราว์เซอร์จริง แล้ว ขึ้น production เรียบร้อย พร้อมงาน LINE auto-login (LIFF) ที่ค้างไว้ด้วย

1 เก็บงานเดิมเข้าที่ก่อน

Merge PR #1 (เก็บเงินก๊วนผ่าน LINE) เสร็จ

รวมงาน “บอท LINE ส่งบิล + รับสลิป + คิวให้เจ้าของยืนยัน” (Phase 1–3) เข้าสาย develop แล้วดันขึ้น master (production) — เป็นฐานให้งานใหม่ต่อยอด

2 ฟีเจอร์ใหม่ — ตั้งค่ายืนยันสลิปรายก๊วน

ทำไมต้องเปลี่ยน

ของเดิมใช้กุญแจตรวจสลิปกลางตัวเดียว ซึ่งมักผูกกับ “บัญชีรับเงิน” บัญชีเดียว — แต่ละก๊วนรับเงินคนละพร้อมเพย์ การตรวจข้ามก๊วนเลยเชื่อถือไม่ได้ จึงเปลี่ยนให้ แต่ละก๊วนถือกุญแจของตัวเอง

ใช้งานยังไง — 2 โหมดต่อก๊วน

  • ยืนยันเอง (ค่าเริ่มต้นทุกก๊วน) — ผู้เล่นส่งสลิป → เข้าคิวให้เจ้าของก๊วนกดยืนยันเอง ฟรี ไม่ต้องตั้งอะไร
  • ใช้กุญแจของก๊วนเอง (BYOK) — ก๊วนสมัคร SlipOK หรือ EasySlip แล้วใส่กุญแจ → ระบบตรวจสลิป + ยืนยันให้อัตโนมัติ

ความปลอดภัยของกุญแจ ตรวจแล้ว

  • กุญแจเก็บใน ตารางแยกเฉพาะ ที่ฝั่งผู้ใช้เข้าไม่ถึง — ไม่เคยถูกส่งไปหน้าเว็บ ไม่ถูกบันทึกใน log
  • หน้าตั้งค่าโชว์กุญแจเป็น •••••••• เท่านั้น (รู้แค่ว่า “ตั้งแล้ว”) จะเปลี่ยนต้องพิมพ์ใหม่
  • เปลี่ยนกลับเป็นโหมดยืนยันเอง → กุญแจถูกลบทิ้งทันที ไม่เก็บค้าง

เบื้องหลังที่ลงมือทำ

เพิ่มคอลัมน์ตั้งค่า + ตารางเก็บกุญแจในฐานข้อมูล, แก้ตัวตรวจสลิปให้รับค่ารายก๊วนแทนอ่านจาก env กลาง, แก้ webhook ให้เลือกโหมดตามก๊วน, ทำหน้าตั้งค่า (เมนูเลือกโหมด + ฟอร์มกรอกกุญแจ) พร้อมรองรับ 2 ภาษา (ไทย/อังกฤษ) — แบ่งงานให้ทีม agent ทำขนานกัน

3 ตรวจคุณภาพก่อนปล่อย (ship-check)

หาบั๊ก → แก้ → ทำให้สะอาด → ทดสอบจริง

  • เจอจุดสำคัญ 1 จุด — ถ้าผู้เล่นมีบิลค้างหลายก๊วนพร้อมกัน สลิปอาจถูกยืนยันผิดก๊วน → แก้แล้ว (บังคับเข้าโหมดยืนยันเองเมื่อมีหลายก๊วน ไม่ตัดสินเองให้พลาด)
  • กันความผิดพลาด — เพิ่มเงื่อนไขชื่อผู้รับต้องยาวพอ กันสลิปผิดบัญชีหลุดผ่าน
  • เก็บความเรียบร้อย — ปรับปุ่ม/ข้อความให้ตรงมาตรฐานโปรเจกต์ + ลดการอ่านฐานข้อมูลซ้ำซ้อนในการบันทึกค่า

ทดสอบบนเบราว์เซอร์จริง ผ่าน

จำลองเจ้าของก๊วนเข้าหน้าตั้งค่าจริง → เลือกโหมดใช้กุญแจเอง + ใส่กุญแจ → บันทึก → ตรวจฐานข้อมูลว่าเก็บถูก → สลับกลับโหมดยืนยันเอง → กุญแจถูกลบ → ไม่มี error และ ลบข้อมูลทดสอบออกหมด ไม่เหลือค้างใน production

4 แก้ปัญหา CI ตอนจะรวมโค้ด

ระบบตรวจอัตโนมัติขึ้นแดง — แต่ไม่ใช่ความผิดของฟีเจอร์

ตอนจะรวมโค้ด ระบบ CI ติดที่ขั้นติดตั้งไลบรารี เพราะไฟล์ล็อกเวอร์ชัน (lockfile) เพี้ยนจากการที่งาน LINE auto-login รวมเข้า develop ก่อนหน้า — แก้ด้วยการดึง develop ล่าสุดมารวม + สร้างไฟล์ล็อกใหม่ให้ครบ → CI กลับมาเขียว
